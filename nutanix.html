Helping Nutanix build a unified storage access platform

In the summer of 2024 I worked with the amazing design team at Nutanix that helped me challenge myself on multiple levels. My work there included pitching ideas to decision makers like the Product Manager, regular syncs with lead developers, and project discussions with other designers.

The outcome was a balanced and simple experience for businesses to access their cloud and private storage via a single shared URL. enhancing nutanix's hybrid multi cloud strategy.

........

Chapter 1: Understanding Nutanix and data centers

In the begining of my internship, I enaged in frequent back and forth with key stakeholders of the project to understand the context, stakeholder's perspectives, and build rapport.

Have you ever wondered what kind of data centers does the military use? What about large established businesses, hospitals, insurance companies? In a statistic published by Cloudzero, over 80% of organizations use a mix of private and public cloud, for business resiliency, efficiency and its cost effectiveness. This mix, called a hybrid multi cloud setup, lets businesses access the best of both worlds.

Nutanix's primary offering is the Prism operating system, designed to operate on private data centers making them as seamless as public data centers. (a.k.a. "the cloud")

Objects Storage & Federations
In data centers, unorganized data like videos and images are stored in so called data lakes. These data lakes implement the Objects Storage System, that works like a valet service. you give to and receive data from the object's manager, without having to worry about where it gets stored iside the lake.







✪ Objects store
Large amounts of physically-connected storage forms an objects store.

✪ Data centers
Each object store in a data center has its own access URL.

✪ Federated URLs
A federated URL provides a single access point for all of its connected URLs.







Buckets
Finally, an Object store can be divided into logical sections for Access control, ease of use, etc. In a real life scenario, an application would ping multiple buckets located in different Object stores to store and access data.


The problem? Federation as a feature only existed for private data centers.

Chapter 2: Understanding the users

Now with the goal of making federations a true hybrid multi cloud feature, I conducted a second round of stakeholder analysis. This helped me identify user archetypes and their associated roles. Futher reading through the PRD gave me an idea of the scope and future plans for the feature.

PC admin
IT admins with access to admin section of Prism OS. Sets up Object Stores and manage physical infrastructure. Creates users and their permissions, access and roles.

Sub admin
Managers that ensure smooth communication between PC admin and End user. Creates buckets in Objects Stores and shares with End user.

End user
Engineers working on applications. Asks the sub admin to create relevant buckets. Pings the federation with get/put requests.

How do they work?
Suppose the end user is working on an application that needs to access a particular bucket in an Object Store. The Object Store is not a member of the Federation that the app runs on, so the sub admin would requests the PC admin to add it to the Federation. Then, the sub admin creates the bucket and shares it to the end user.

Chapter 3: Ideas and the breakthrough

The biggest challenge we faced was deciding on an Object Store equivalent for external storage. Each cloud provider has a different approach to storage infrastructure.

Understanding the black box
Let's simplify everything with the help of a metaphor. You have files on your laptop, as well as on google drive. If you wanted to create a central access point for your relevant files on your laptop, would you add,


  1. Just the relevant folders, individually
  2. A single parent folder that contains all relevant sub folders inside
  3. The entire google drive account along with its permissions

Slap role-based access control on top of it for three different users and you have the problem at hand.

Deciding the approach
I came up with six different approaches to the solution, which we narrowed down to three with the help of my mentor. Using the well developed design library, I was able to quickly design major flows for these three approaches, to present to the stakeholders.

Adding external storage locations hit the right balance & achieved consensus amongst stakeholders.

Chapter 4: The final showdown

With the approach set, I covered all aspects of the setup using the CRUD framework - Create ✦ Read ✦ Update ✦ Delete. Let us look at one of the major flows, as discussed earlier, of adding a bucket to a federation.

Prism OS's Federation admin page
The page shows a summary of the federation, a table of member Object Stores, and an empty table to add external endpoints(storage points).

The PC admin now clicks on add a new External Endpoint

Add external endpoint
Next a form appears to enter the details of the external endpoint, as provided by the Sub admin. The illustration dynamically updates with the form.

After hitting save, the new endpoint appears on the federation page. With this, The PC admin role is over.

Sub admin Federation view
Next, the Sub admin will log onto their view of the Federation, to create a new bucket.

The Sub admin now selects the option to add an external bucket.

Add external bucket
A new form appears, this time for the details of the bucket.

A new bucket appears in the list of buckets. This bucket needs to be shared with the End user.

The Sub admin selects the bucket and clicks share. In the new page, they enter the details of the End user to share to. With this, the Sub admin role is over.

End user Federation view
Finally, the End user is able to log on to their version of federation and access the bucket shared to them!

With this, ends the first and the most important flow in the entire feature.

Error States
Along with the flows, I also identified and designed the major error states, and collaborated with the content team to write the error messages.

Accessibility
Further, I performed a small accessibility audit on the designs, which included identifying screen reader order, and alt text for buttons.

Handoff
After over nine weeks of work, I was finally able to handoff the high fidelity designs, where I highlighted every change made for future reference and ease of understanding.

Learnings

• Designers are users advocates in between Developers who might want to ship the easiest solution and PM who might want the most marketable features.

• User is not always the customer sometimes the person buying the product might be a Chief Officer of the company and the person using it an understaffed IT admin.

• Get frequent feedback on rough ideas instead of polished outputs people are less likely to criticize well polished outputs.
